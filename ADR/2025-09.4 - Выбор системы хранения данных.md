
**Дата:** 20 сентября 2025 г.

**Статус:** Принято

**Контекст:**

Нам нужна надежная и масштабируемая база данных для нашей системы фриланс биржи. 
Выбор базы данных может влиять на производительность, масштабируемость и безопасность нашего решения.

**Рассмотренные варианты:**

1. **PostgreSQL:** Объектно-реляционная система управления базами данных.
2. **MySQL:** Популярная реляционная система управления базами данных.
3. **MongoDB:** Документоориентированная система управления базами данных.
4. **Cassandra:** Распределенная система управления базами данных.

**Решение:**

Выбран MySQL как основная СУБД и Cassandra для мессенджера

**Обоснование:**

- **Надежность:** MySQL известен своей стабильностью и надежностью, даже при работе с большими объемами данных.
- **Расширяемость:** MySQL поддерживает расширения, что позволяет добавлять новые возможности по мере необходимости (шардирование).
- **Оптимизация запросов:** Отличный планировщик запросов, который оптимизирует сложные запросы.
- **JSON поддержка:** Встроенная поддержка JSON позволяет работать с неструктурированными данными.
- **Сообщество:** Большое и активное сообщество, множество доступных материалов и ресурсов для обучения.
- **Опыт:** Опыт разработки на MySQL у разработчиков
- **Быстрая запись:** Касается Cassandra, т.к. в мессенджере операций записи больше, чем чтений
- **Репликация:** Возможность реплецировать данные как в MySQL так и в Cassandra
  
**Последствия:**

- Шардирование: Учесть виртуальное шардирование с последующим горизонтальным
- Две технологии: Команде потребуется экспертиза как в MySQL, так и в Cassandra, что может увеличить затраты на обучение и найм.
- Сложность разработки: Придется реализовывать и поддерживать логику работы с двумя различными базами данных, включая транзакции, миграции и мониторинг для каждой из них.
- Согласованность данных: В системе появляется два источника истины. Необходимо тщательно проектировать интеграцию между сервисами, работающими с разными БД, чтобы обеспечить согласованность данных.
- Мониторинг: Требуется настройка раздельного мониторинга производительности и здоровья для двух различных систем баз данных.
- Резервное копирование: Стратегии бэкапа должны быть разработаны и реализованы отдельно для каждой СУБД, учитывая их архитектурные особенности.
- Операционные расходы: Поддержка двух различных систем баз данных увеличивает операционные расходы и сложность управления инфраструктурой.
