**ADR: 2025-09.2 - Выбор технологии серверной части фриланс-биржи**

**Статус:** Принято

**Контекст:**
Разработка высоконагруженной фриланс-биржи требует выбора технологии, способной обрабатывать тысячи одновременных пользователей при обеспечении доступности и устойчивости к разделению (AP в CAP-теореме).

**Решение:**
Выбран **Go** как основной язык разработки серверной части.

**Рассмотренные варианты:**

**Go:**
- Высокая производительность компилируемого языка
- Эффективная модель конкурентности (горутины)
- Низкое потребление памяти и CPU
- Богатая стандартная библиотека
- Простота развертывания и поддержки

**Java:**
- Мощная enterprise-экосистема
- Высокая производительность JVM
- Spring Boot для быстрой разработки
- Сложная модель конкурентности
- Высокое потребление памяти

**Python:**
- Быстрая разработка
- Богатые фреймворки (Django, FastAPI)
- Проблемы с GIL для многопоточности
- Низкая производительность для CPU-bound задач

**C++:**
- Максимальная производительность
- Полный контроль над ресурсами
- Высокая сложность разработки
- Длинный цикл разработки

**Обоснование выбора Go:**

**Производительность:**
- Обработка 50K+ одновременных соединений
- Нативная компиляция в бинарные файлы
- Эффективное использование ресурсов сервера

**Масштабируемость:**
- Идеален для микросервисной архитектуры
- Простое горизонтальное масштабирование
- Минимальные overhead при репликации

**Разработка:**
- Короткий цикл разработки
- Простота поддержки и рефакторинга
- Строгая типизация для надежности

**Экосистема:**
- Встроенная поддержка HTTP/2, JSON, protobuf
- Богатые библиотеки для работы с сетью
- Отличная документация и сообщество

**Последствия:**

**Положительные:**
- Высокая производительность под нагрузкой
- Экономия ресурсов серверов
- Быстрое развертывание и масштабирование
- Надежность и стабильность работы

**Отрицательные:**
- Меньше enterprise-решений по сравнению с Java
- Ограниченные фреймворки для сложных бизнес-процессов
- Необходимость написания большего количества кода по сравнению с Python
